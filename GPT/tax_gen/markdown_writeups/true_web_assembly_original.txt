<h2>Solution</h2>

<p>The challenge is to attack the latest version of <a href="https://asm32.info/fossil/asmbb/index">AsmBB</a>, a web-based message board implemented entirely in x86 assembly. The provided Dockerfile builds the <code>asmbb</code> engine using the source files from the <code>asmbb</code> and <code>freshlib</code> repositories.</p>

<p>```docker</p>

<h1>Get source files for asmbb</h1>

<p>RUN wget https://asm32.info/fossil/asmbb/tarball/4c91cddaec/asmbb-4c91cddaec.tar.gz -O asmbb.tar.gz &amp;&amp; \
    /bin/bash -c "echo 'b1e621d1ae988b35e836ec9142ccc6ce6cf7c24a090c4d973894770a62fa4ddc asmbb.tar.gz' | sha256sum --check" &amp;&amp; \
    tar -xf asmbb.tar.gz || true &amp;&amp; \
    mv asmbb-* asmbb</p>

<h1>Get source files for freshlib</h1>

<h1>AsmBB uses functions from freshlib</h1>

<p>RUN wget https://fresh.flatassembler.net/fossil/repo/fresh/tarball/6636a57441/Fresh+IDE-6636a57441.tar.gz -O fresh.tar.gz &amp;&amp; \
    /bin/bash -c "echo '5ba395b0e957536bd66abc572414085aab5f2a527d28214881bbba72ec53e00d fresh.tar.gz' | sha256sum --check" &amp;&amp; \
    tar -xf fresh.tar.gz &amp;&amp; \
    mv Fresh* Fresh</p>

<h1>Build the asmbb engine</h1>

<p>RUN lib=/Fresh/freshlib TargetOS=Linux /fasm/fasm -m 200000 /asmbb/source/engine.asm /engine
```</p>

<h2>Gaining XSS</h2>

<p>The forum is the main feature of AsmBB, and the default build uses a custom markdown-like parser called MiniMag. Our goal is to achieve a GET-based XSS on the admin user, and subsequently abuse admin features for RCE.</p>

<figure><img src="../../.gitbook/assets/Screenshot 2023-03-12 at 10.18.45 PM.png" alt=""><figcaption></figcaption></figure>

<p>Let's take a look at the AsmBB <a href="https://asm32.info/fossil/asmbb">source</a>. <a href="https://asm32.info/fossil/asmbb/file?name=source/render2.asm\&ci=4c91cddaec90fb74"><code>render2.asm</code></a> contains a "hash table" of commands used by the templating engine, mapped to their routines.</p>

<p><code>nasm
PHashTable tableRenderCmd, tpl_func,                      \
        'special:',     RenderTemplate.cmd_special,       \
        'raw:',         RenderTemplate.cmd_raw,           \
        'include:',     RenderTemplate.cmd_include,       \
        'minimag:',     RenderTemplate.cmd_minimag,       \   ; HTML, no encoding.
        'bbcode:',      RenderTemplate.cmd_bbcode,        \   ; HTML, no encoding.
        'html:',        RenderTemplate.cmd_html,          \   ; HTML, disables the encoding.
        'attachments:', RenderTemplate.cmd_attachments,   \   ; HTML, no encoding.
        'attach_edit:', RenderTemplate.cmd_attachedit,    \   ; HTML, no encoding.
        'url:',         RenderTemplate.cmd_url,           \   ; Needs encoding!
        'json:',        RenderTemplate.cmd_json,          \   ; No encoding.
        'css:',         RenderTemplate.cmd_css,           \   ; No output, no encoding.
        'equ:',         RenderTemplate.cmd_equ,           \
        'const:',       RenderTemplate.cmd_const,         \
        'enc:',         RenderTemplate.cmd_encode,        \   ; encode the content in html encoding.
        'usr:',         RenderTemplate.cmd_user_encode    \   ; encodes the unicode content of the user nickname for unicode-clones distinction.
</code></p>

<p>We can see this in action in <a href="https://asm32.info/fossil/asmbb/file?name=www/templates/Urban+Sunrise/post_view.tpl\&ci=4c91cddaec90fb74"><code>post_view.tpl</code></a> where the post is rendered. Depending on <code>format</code>, the post content is either parsed with <code>minimag</code> or <code>bbcode</code>, and the final output is rendered as HTML.</p>

<p>```html</p>

<article class="post-text">
  [html:[[case:[format]|minimag:[include:minimag_suffix.tpl]|bbcode:][Content]]]

</article>

<p>```</p>

<p>Although the client-side UI only allows us to write content in the MiniMag format, the POST request to submit the post does include a <code>format</code> parameter.</p>

<p>```http
POST /!post HTTP/1.1
Host: localhost:9032
Content-Length: 917
...
Connection: close</p>

<p>...</p>

<p>------WebKitFormBoundarydKCsA6RKHAepAWPn
Content-Disposition: form-data; name="format"</p>

<p>0
------WebKitFormBoundarydKCsA6RKHAepAWPn
Content-Disposition: form-data; name="source"</p>

<p>[http://example.com][My link] 
------WebKitFormBoundarydKCsA6RKHAepAWPn--
```</p>

<p>When set to 1, the <code>format</code> parameter allows us to use the <a href="https://en.wikipedia.org/wiki/BBCode">BBCode</a> parser instead. This uses the <code>bbcode</code> command, which calls the <code>.cmd_bbcode</code> routine.</p>

<p>```nasm
.cmd_bbcode:
; here esi points to ":" of the "bbcode:" command. edi points to the start "[" and ecx points to the end "]"</p>

<p>locals
  BenchVar .bbcode_time
endl</p>

<pre><code>    BenchmarkStart .bbcode_time

    stdcall TextMoveGap, edx, ecx
    stdcall TextSetGapSize, edx, 4
    mov     dword [edx+ecx], 0
    add     [edx+TText.GapBegin], 4
    inc     [edx+TText.GapEnd]              ; delete the end "]"

    stdcall TextMoveGap, edx, edi
    add     [edx+TText.GapEnd], 8

    stdcall TranslateBBCode, edx, edi, SanitizeURL

    ...
</code></pre>

<p>```</p>

<p>Since the BBCode parser was a <a href="https://board.asm32.info/the-latest-update-of-this-forum.258/">newer</a> parser introduced after MiniMag, and isn't enabled by default, we thought this would be the best place to start looking for parser vulnerabilities.</p>

<p>The <code>TranslateBBCode</code> routine from <a href="https://fresh.flatassembler.net/fossil/repo/fresh/artifact/0457fbe206805cbe"><code>bbcode.asm</code></a> (found in <a href="https://fresh.flatassembler.net/fossil/repo/fresh">FreshLib</a>) is then used to parse the BBCode content. Here we see a table of supported BBCode tags.</p>

<p>```nasm
PHashTable tableBBtags, tpl_func,                      \
        'b',       tagStrong,                          \
        '*',       tagListItem,                        \
        'i',       tagEm,                              \
        'u',       tagUnderlined,                      \
        's',       tagDel,                             \
        'c',       tagInlineCode,                      \
        'url',     tagURL,                             \
        'img',     tagImg,                             \
        'quote',   tagQuote,                           \</p>

<pre><code>    ...
</code></pre>

<p>```</p>

<p>BBCode is an old markup language that has a rather simple syntax. Tags are enclosed by square brackets, and some tags can have attributes, such as the following URL tag:</p>

<p><code>bbcode
[url=https://example.com]My link[/url]
</code></p>

<p>The main loop of the parser is found at <code>.loop</code>. For each character, the logic goes:</p>

<ul>
<li>if the end of the text has been reached, exit the loop</li>
<li>if it is a newline or space character, skip it</li>
<li>if it is <code>[</code>, process the tag at <code>.start_tag</code></li>
<li>if it is the start of an emoji, process the emoji</li>
</ul>

<p>```nasm
.loop:
        mov     ecx, [edx+TText.GapEnd]
        cmp     ebx, [edx+TText.GapBegin]
        cmovb   ecx, [edx+TText.GapBegin]
        sub     ecx, [edx+TText.GapBegin]
        add     ecx, ebx
        cmp     ecx, [edx+TText.Length]
        jae     .end<em>of</em>text</p>

<pre><code>    movzx   eax, byte [edx+ecx]

    test    al, al
    jz      .end_of_text

    cmp     al, $0d
    je      .new_line

    cmp     al, $0a
    je      .new_line

    cmp     al, $20
    jbe     .next           ; skip all whitespace

    ...
</code></pre>

<p>.paragraph_ok:</p>

<pre><code>    cmp     al, "["
    je      .start_tag
</code></pre>

<p>; here check for emoticons</p>

<pre><code>    cmp     al, $f0         ; emoji?
    jb      .continue

    ...
</code></pre>

<p>```</p>

<p>Otherwise, we go to <code>.continue</code>, where the character is HTML encoded.</p>

<p>```nasm
.continue:</p>

<p>; html encoding from here</p>

<pre><code>    test    al, al          ; all values &gt; 127 are unicode and should not be encoded.
    js      .next

    movzx   eax, byte [tbl_html+eax]
    test    al, al
    jz      .del_char
    jns     .next           ; the same as above

    lea     esi, [eax+tbl_html]     ; the address of the replacement string.
    lodsb
    movzx   ecx, al         ; length
</code></pre>

<p>; insert the replacement html encoding from esi
        stdcall TextMoveGap, edx, ebx
        stdcall TextSetGapSize, edx, ecx
        inc     [edx+TText.GapEnd]      ; delete the previous char.</p>

<pre><code>    mov     edi, [edx+TText.GapBegin]
    add     edi, edx
    add     [edx+TText.GapBegin], ecx
    add     ebx, ecx

    rep movsb
    jmp     .loop
</code></pre>

<p>```</p>

<p>Notice that unless the current character is part of an emoji or part of an opening/closing tag, we will reach the HTML-encoding logic. This is done through a simple text substitution that sanitizes angle brackets, quotes, and ampersands.</p>

<p><code>nasm
HtmlEntities tbl_html,        \
  $09, $0d,                   \
  $0a, $0a,                   \
  $0d, $0d,                   \
  '&lt;', '&amp;lt;',                \
  '&gt;', '&amp;gt;',                \
  '"', '&amp;quot;',              \
  "'", '&amp;apos;',              \
  '&amp;', '&amp;amp;'
</code></p>

<p>Since everything outside the opening/closing tag are HTML-encoded, let's take a closer look at the tag-processing logic. When a tag is matched, a string substitution is performed based on the table below.</p>

<p>```nasm
...</p>

<p>tagImg          onetag <txt '<img class="block"', HTML_IMG_ATTR, 'alt="'>, txt '" src="',  txt '" />',         fBlockTag  or fDisableTags or fURLContent
tagInlineImg    onetag <txt '<img class="inline"', HTML_IMG_ATTR,'alt="'>, txt '" src="',  txt '" />',         fInlineTag or fDisableTags or fURLContent
tagSize         onetag txt '<span style="font-size:',               txt '">',       txt '</span>',      fInlineTag
tagColor        onetag txt '<span style="color:',                   txt '">',       txt '</span>',      fInlineTag
tagEmail        onetag txt '<a href="mailto:',                      txt '">',       txt '</a>',         0</p>

<p>...
```</p>

<p>The 2nd, 3rd, and 4th columns correspond to the start of the tag, end of the attribute, and end of the tag respectively. For instance, the following markup</p>

<p><code>bbcode
[email=example@example.com]Click Here[/email]
</code></p>

<p>becomes</p>

<p><code>
&lt;a href="mailto: + example@example.com + "&gt; + Click Here + &lt;/a&gt;
</code></p>

<p>The attribute value and the content in between the opening/closing tags are processed separately from the tag itself, and are thus subject to HTML-encoding. If there's any parsing bug to be found, it would probably have to be while parsing the tag.</p>

<p><em>What if we just don't close the tag?</em></p>

<p>Since the tag isn't being encoded while it is processed, there might be an edge case where the unencoded content is reflected in the absence of a closing <code>]</code>.</p>

<p><em>Voilà</em>, the following markup</p>

<p><code>bbcode
[email=&lt;img src=x onerror=alert() 
</code></p>

<p>translates to</p>

<p><code>html
&lt;a href="mailto:&amp;lt;img src=x onerror=alert() "&gt;&lt;img src=x onerror=alert() &lt;/a&gt;
</code></p>

<p>which when rendered on a browser, pops an alert!</p>

<h2>Gaining RCE</h2>

<h2>Putting It All Together</h2>

<p>Here's the final exploit that we will serve to the admin. Here, I used a first-stage payload to keep the exploit small, but serving the whole exploit in one payload would work as well.</p>

<p><code>javascript

</code></p>

<p>is converted to base64 and eval-ed:</p>

<p>{% code overflow="wrap" %}
<code>

{% endcode %}</p>

<p>which then executes the RCE payload:</p>
